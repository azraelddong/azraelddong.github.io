2、关闭防火墙/iptables

systemctl stop iptables && systemctl disabled iptables

hostnamectl set-hostname lixd && bash

Containerd和Docker

Docker包含Containerd，containerd专注运行时的容器管理，而Docker出来管理容器外，还可以构建容器。

k8s在1.24版本之后，弃用docker。

调用链：

Docker作为k8s容器运行：

Kubelet –> docker shim –> docker –> containerd

Containerd作为k8s容器运行：

kubelet -> CRI -> containerd

🌈K8S

组件：

  • api server： 认证授权策略，统一入口，

  • kubelet
  • kube-proxy：service负载均衡代理，生成iptables规则，ipvs
  • coredns
  • calico：网络插件，跨节点访问，网络策略

pod：最小单元

存储

  • empty dir
  • hostPath
  • NFS：单节点故障问题，不是分布式存储的
  • PV/PVC
    • PV是一个存储块
    • PVC是存储请求，与PV绑定
    • 回收策略：Retain、Recycle（不推荐）、Delete

Pods

namespace

健康探测：

  • 存活探测：监测pod是否处于运行状态，探测失败，重启容器
  • 就绪探测：监测pod是否已经准备好接受流量，探测失败，不将流量转发到该容器
  • 启动探测：监测pod是否启动成功，与存活探测不同的是，只会执行一次。当探测失败，将自动重启容器

为什么要有四层代理？？

1、Pod每次删除新增后，ip是动态变化的，使用service四层代理，这样，ip不管怎么变化，都不会影响服务访问。

2、Pod的ip在k8s集群之外，是无法访问的，所以需要使用service代理，才可以在k8s集群之外才能访问。

Kube-proxy：在创建service的时候，会把对应的servcie和对应的Pod的ip保存到iptables或者ipvs规则里。

Coredns：将域名解析成ip

Service原理：

k8s创建service的时候，同时会创建与service同名的entpoint对象，当pod地址发生变化，entpoint也会随之变化，service接收请求后，就会通过entpoint，转发到对应的pod容器，

至于转发到具体哪个pod容器，由负载均衡kube-proxy决定。

service的名称：<service.name>..svc.cluster.local

k8s中有三个网络：

  • 物理机ip
  • pod的ip
  • service的ip

NodePort、ClusterPort和ExternalName的service的区别？

NodePort类型的service，映射的端口后，可以在集群外访问

ClusterPort类型的service，映射的端口后，只能在集群内访问

ExternalName类型的service，可以跨命令空间访问，相当于做软链

Istio流量治理

Istio是什么？

连接、安全加固、控制和观察服务的开放平台。

🌈Istio的四个特性：

1、连接：智能控制服务之间的调用流量，能够实现灰度发布，A/B测试和蓝绿部署等功能

2、安全加固：自动为服务间调用提供认证、授权和加密

3、控制：保证资源在消费者之间公平分配

3、观察：查看服务运行期间的各个参数，比如日志、监控和链路调用等

4、帮助微服务分层解耦，解耦后的proxy层可以更加专注提供基础架构能力：

（1）服务发现

（2）负载均衡

（3）故障恢复

（4）服务监控

（5）A/B测试

（6）灰度发布

（7）限流限速

（8）访问控制

断路器

服务熔断：达到一定流量之后，服务直接熔断，不再提供服务

服务降级：提高用户体验，当有大量流量过来的时候，返回友好页面或者提示
